Fix a background-jobs dispatch wedge: a worker only re-announced free forked/inline capacity on the exact `size === cap - 1` transition, but the main removes a worker from its ready set on every dispatch and only re-adds it on a fresh "ready" — so a single missed or dropped signal left the worker out of the ready set and silently froze dispatch cluster-wide under bursty high-concurrency load. Workers now re-announce readiness on every completion that leaves them below capacity, and re-announce spare capacity on each heartbeat as a safety net, so no single lost message can wedge the queue.
