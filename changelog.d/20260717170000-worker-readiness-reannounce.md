Fix a background-jobs dispatch wedge: a worker only re-announced free forked/inline capacity on the exact `size === cap - 1` transition, but the main removes a worker from its ready set on every dispatch and only re-adds it on a fresh "ready" — so a single missed signal left the worker out of the ready set and silently froze dispatch cluster-wide under bursty high-concurrency load. Workers now re-announce readiness on every completion that leaves them below capacity. Each re-announce corresponds to a genuinely freed slot, so it never advertises capacity that a still-in-flight handoff will consume.
