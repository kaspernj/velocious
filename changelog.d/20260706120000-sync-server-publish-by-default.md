Publish server-side sync changes to the feed and broadcasts by default: `SyncPublisher` (`src/sync/sync-publisher.js`) is the server mirror of the client's track-by-default mutation tracking. Server models declaring `static sync = {publish: {serialize, eventId?, broadcasts?, operations?, resourceType?}}` automatically write every committed server-side create/update (destroys opt-in via `operations`) to the sync change feed — a model-backed Sync-row upsert with server re-sequencing, shared with the replay service's persistence through `src/sync/sync-change-fanout.js` — and fan out the declared broadcasts, so server-origin changes (like an event PIN saved on the server) reach the change feed and every device without manual upsert/broadcast helpers. Started automatically at server boot via `SyncPublisher.startFromConfiguration(configuration)` (no-op when no registered model declares publish); `publish: false` opts a model out. Payloads are snapshotted at mutation time, publishing defers through the connection's `afterCommit` hook (rollbacks never publish), and post-commit failures report loudly without poisoning the afterCommit chain. Replayed device mutations never double-publish: the routed replay apply marks its written records through the new server suppression API — `markServerApply(record)` and `withoutPublishing(callback)` in `src/sync/sync-publish-suppression.js` — which importer/backfill code applying already-synced data can use the same way. The sync client now also tolerates (and ignores) the server-side `publish` key in shared `static sync` declarations.
