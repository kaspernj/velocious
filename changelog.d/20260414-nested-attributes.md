- Add Rails-style `nestedAttributes` support for frontend-model `save()`. Nested writes require BOTH a model-level opt-in (`Project.acceptsNestedAttributesFor("tasks", {allowDestroy: true, limit?})` on the backend `Record` subclass, matching Rails' `accepts_nested_attributes_for`) AND a resource-level permission (`static nestedAttributes = {tasks: {allowDestroy?, limit?, rejectIf?}}` on the frontend-model resource). The backend throws if the model has not opted in, so a resource can never silently expose nested writes on a model that doesn't declare them safe. Resource options may restrict but not loosen the model's declared policy (e.g. a resource can't enable destroy if the model doesn't). The client walker on `save()` collects dirty children (new records, records marked with `markForDestruction()`, records with changed attributes, records with dirty descendants) and ships them in a `nestedAttributes` payload. The backend unpacks the payload, authorizes each child against its own resource's abilities, and applies destroys â†’ updates â†’ creates in order, all inside a single DB transaction.
- `FrontendModelBase` gains `markForDestruction()` and `markedForDestruction()` for queuing destroy-on-next-parent-save.
- `FrontendModelResourceConfig` typedef updated to include `attributes`, `relationships`, and `nestedAttributes`, matching the generator output.
- Fix: `setAttribute` no longer clears every cached relationship on any attribute change. It now only invalidates a `belongsTo` cache when the attribute matches that relationship's foreign key (declared or inferred from the relationship name). This preserves preloaded children and nested-save state during parent edits.
- Add api_maker-style `permittedParams(arg)` method on `FrontendModelBaseResource`. Returns a permit spec `{attributes: string[] | null, nestedAttributes: Record<string, {allowDestroy?, limit?, rejectIf?}>}` that declares which attributes and nested relationships are writable for the current request. `arg` carries `{action, params, ability, locals}` so subclasses can restrict writes by user role, action, or request context. Default implementation reads static `nestedAttributes` and leaves `attributes: null` (legacy setter-existence behavior) for backward compatibility.
