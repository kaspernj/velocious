- **Breaking:** `permittedParams(arg)` now returns a Rails/api_maker-style flat array that mixes attribute-name strings with `{<relationshipName>Attributes: [...]}` objects, e.g. `["firstName", "lastName", {contactAttributes: ["email", {detailsAttributes: ["detail"]}]}]`. The previous shape `{attributes: [], nestedAttributes: {...}}` is gone. The default implementation returns `[]` — nothing is permitted — so resources must override to enable any writes.
- **Breaking:** `nestedAttributes(arg)` resource method and its `static` predecessor have been removed. Nested relationships are declared inline in `permittedParams(arg)` via the `{<relationshipName>Attributes: [...]}` shape. The parent's permit spec governs what attributes its nested children can write. Policy options (`allowDestroy`, `limit`, `rejectIf`) stay on the model side via `acceptsNestedAttributesFor`. To allow `_destroy: true` for a nested relationship, include `"_destroy"` in its permit array **and** set `allowDestroy: true` on the model declaration.
- Generator now extracts nested relationship names from each resource's `permittedParams()` (calling it with no arg) and emits them into the generated frontend-model `resourceConfig().nestedAttributes` as `{relationshipName: {}}` so the client `save()` walker knows which relationships to ship.
