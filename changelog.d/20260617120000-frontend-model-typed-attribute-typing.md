Generated typed-attribute frontend models (`FrontendModelBase<XAttributes, XCreateAttributes, XUpdateAttributes>`) now typecheck against the framework's own types. The frontend-model generic constraints (`FrontendModelClass`, `FrontendModelBase`, the relationship helpers, and the `FrontendModelRelationship` union) defaulted to a concrete `Record<string, FrontendModelTransportValue>`, which — by TypeScript invariance — made every `Model.findBy()`/relationship access on a generated typed model fail in consuming apps. The defaults are now permissive so any generated subclass satisfies them, while the static query methods' own `@template T` still captures the precise calling class for return types.

The generator's model-accessor attribute inference now unwraps `Promise<T>` to `T` (a frontend attribute holds the serialized resolved value, not a Promise) and falls back to `any` when the inferred type references a backend-only typedef that can't resolve in the generated frontend model.

Custom command entries in `collectionCommands`/`memberCommands` may now be a `{name, args?, returnType?}` object (alongside the plain camelCase-name string form). Declared `args: [{name, type}]` generate named, typed method parameters mapped positionally into the command payload; a declared `returnType` types the method `Promise<thatType>` instead of the generic `Promise<Record<string, ?>>`.
