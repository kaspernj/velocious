# Changelog

- Add an optional `backgroundJobs.jobTimeoutMs` (or `VELOCIOUS_BACKGROUND_JOBS_JOB_TIMEOUT_MS`) wall-clock backstop for `"forked"` job runners. A forked job still running after the timeout is terminated (SIGTERM, then SIGKILL after the reaping grace) and reported `failed`, so a genuinely-hung runner can no longer pin a worker — and its whole-app boot and database connections — indefinitely (notably a retired-release worker draining after a deploy). It is a coarse safety net, not per-job tuning: it applies to every forked runner, so set it well above the longest legitimate forked job. Omit it, or set `null`/`<= 0`, to disable (the default, preserving the prior unbounded behavior); an explicit config value wins over the environment variable, and a value beyond Node's timer range (or `Infinity`) is clamped/disabled rather than coerced to ~1ms. `"inline"` jobs are not covered — they share the worker's process and cannot be killed without killing the worker.
