# Changelog

- Sync clients can now ride one shared app-lifetime websocket for everything. Configure `sync.client.websocketUrl` (the framework builds and owns one reconnecting `VelociousWebsocketClient`, connected on first use) or `sync.client.websocketClient` (the low-level form — pass the same instance the frontend-model transport uses so a single socket carries frontend-model traffic and sync). `syncClient().syncConnection()` resolves the shared connection (or null). The realtime bridge rides it without owning its lifecycle: `unsubscribeRealtime()` now closes only its channel subscriptions and leaves the socket open, while the deprecated per-cycle `sync.client.realtime.createClient` path (build + disconnect its own client each subscribe cycle) keeps working unchanged when no shared connection is configured.
- Added the server-enumerated user scope: `syncClient().subscribeUserScope()` declares a user scope (empty conditions) for every pullable synced resource type, subscribes their `velocious-sync` channels, and pulls — the client subscribes once with just its token and the server decides membership. The framework sync channel authorizes each empty-conditions scope through the app sync resource's existing `authorizeChanges` and re-checks record access per delivery at broadcast fan-out through the sync resource's new `changeDeliverable`, which reuses the app's `scopeChangesQuery` ability scoping (empty-conditions scopes fall back to ability scoping, with per-scope cursor continuity for pulls). Two subscribers with disjoint access each receive only their own changes over the one connection. `unsubscribeUserScope()` deactivates the scopes and closes the subscriptions without disconnecting the shared connection (sign-out drops subscriptions, keeps the socket), and is idempotent and single-flighted like `subscribeRealtime()`.
