Per-event frontend-model websocket broadcast authorization now runs in the subscriber's own tenant scope. The channel re-authorizes each broadcast record with `frontendModelAuthorizedQuery("find")`, but it ran under whatever ambient tenant context the publisher left behind. For a multi-tenant record mutated outside the subscriber's own request (a relay endpoint, a background job, another user's action), that ambient tenant lacked the subscriber's per-record access flags, so the authorization query found nothing and the `create`/`update` event was silently dropped — the subscriber never received live updates for rows it can legitimately read. The channel now re-resolves the tenant from the event record id plus the subscriber's request and runs the per-event authorization/filter/projection queries inside `runWithTenant(...)` (a no-op when no tenant resolves, preserving non-multitenant behavior).
