`Record.withAdvisoryLock` and `Record.withAdvisoryLockOrFail` now accept an optional `holdTimeoutMs`. If the callback runs longer than it, the lock is released and an `AdvisoryLockHoldTimeoutError` (newly exported) is thrown, so a hung holder can't block other sessions indefinitely. Default behaviour is unchanged (no hold timeout unless requested). The callback is not cancelled — `holdTimeoutMs` is a safety net, not cancellation; pass an `AbortSignal` to the work if it needs to stop.
