Background job workers no longer drop a job's terminal status when persisting it fails transiently. `BackgroundJobsStatusReporter#reportWithRetry` previously threw immediately on a `BackgroundJobUpdateError` (the `job-update-error` the jobs-main process answers when its `store.markCompleted`/`markFailed` throws), so a transient DB error while recording a completion — a deadlock, a connection reset, a lock-wait timeout, or jobs-main's cold connection pool right after a deploy restart — permanently stranded the job in `handed_off`. For a `max_concurrency: 1` job such as a build/queue planner, one stranded row then blocked every future run of that job until an operator intervened. `job-update-error` is only ever answered when the DB write throws (all logical rejections answer `job-updated`), so it is always the transient, retryable kind; it is now retried with backoff until the terminal status is persisted, bounded by the optional `maxDurationMs`.
