## Added

- `VelociousDatabaseRecord.withAdvisoryLock(name, callback, {timeoutMs})`, `withAdvisoryLockOrFail(name, callback)`, and `hasAdvisoryLock(name)` static methods for cooperative, connection-scoped advisory locks. Unlike row locks, these do not block readers or writers that are not participating in the same named lock, so they are the preferred way to serialize specific functionality (e.g. a decrement step that must not race against itself) without blocking unrelated traffic against the same row. Implemented natively on MySQL/MariaDB (`GET_LOCK` / `RELEASE_LOCK` / `IS_USED_LOCK`), PostgreSQL (`pg_advisory_lock` / `pg_try_advisory_lock` / `pg_advisory_unlock` with FNV-1a key hashing), and SQL Server (`sp_getapplock` / `sp_releaseapplock`). SQLite has no native advisory lock primitive, so the SQLite base class ships a process-local emulation with a waiter queue, and the Node SQLite driver layers a filesystem lock (mkdir-based lock directories with PID + hostname metadata and stale-lock recovery) on top of that emulation for multi-process mutual exclusion. Web (sql.js) and Expo native inherit the in-process implementation unchanged. Timing out or contending callers get dedicated `AdvisoryLockTimeoutError` and `AdvisoryLockBusyError` instances exported from `database/record/index.js`.
