Resource-routed sync replay: `SyncEnvelopeReplayService` accepts `configuration`, `resourceTypeOverrides`, `ability`, `abilityContext` and `locals` and routes mutations without a matching applyHandler through the registered frontend-model resource classes (applySync escape hatch, authorizeSyncMutation, ability-scoped findSyncRecord, schema-normalized assign+save updates, save-then-check membership creates, delete/missing-record behaviors and afterSyncApply extras). Client-safe apply failures fail only their own sync (`{id, syncState: "failed", reason, message}`) and keep the batch going. `SyncResourceBase#buildReplayService` plumbs ability/context/configuration/locals under `replayServiceArgs()` and `replayServiceClass()` defaults to `SyncEnvelopeReplayService`.

`writableAttributes` moved up to `FrontendModelBaseResource`: declaring the schema derives the default `permittedParams` (camelCase keys) and runs create/update normalization automatically (new normalizer `keyCase: "attribute"` option). Schema entries may be `true` or partial objects — `type`, `maxLength` and `required` are inferred from the model's column metadata via `resolvedWritableAttributes()` (explicit fields win). New normalizer types/options: `boolean`, `integer` (`min`/`max`), `float`, `uuid`, `rawOrNull`, `ignored`, date `invalid: "null"` and string `trim: false`.

Validation messages now route through a translated message layer (`velocious.errors.messages.*` msgIDs via `configuration.getTranslator()`, Rails-style defaults such as "can't be blank"): the presence/uniqueness model validators, the sync attribute normalizer's derived messages, and routed replay model-validation failures (surfaced per sync as `reason: "validation-error"` with the translated message). Per-attribute `*Message` schema strings remain as a legacy escape hatch. Also adds `resolveFrontendModelResourceClass` for registry-based resource-type resolution honoring `modelName` overrides.
