`Record.withAdvisoryLock` and `Record.withAdvisoryLockOrFail` now acquire and release advisory locks through a dedicated lock connection when a positive `holdTimeoutMs` is configured. Calls without a positive `holdTimeoutMs` keep using the caller's application connection to avoid extra connection overhead. The dedicated hold-timeout path lets Velocious release the lock before raising `AdvisoryLockHoldTimeoutError`, even when the callback's own database work is still hung.
